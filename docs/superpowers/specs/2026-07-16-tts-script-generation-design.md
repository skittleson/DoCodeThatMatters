# TTS Script Generation Step — Design

**Date:** 2026-07-16
**Status:** Approved, ready for planning

## Problem

Audio is currently generated by feeding a markdown-to-plain-text conversion
(`docs/<slug>/index.tts`) straight into Kokoro TTS. The result sounds wrong: the
narrator reads raw URLs aloud and stumbles over artifacts left over from the
markdown-to-text conversion. There is no step that turns the post into something
a human would actually *say*.

## Outcome

Insert an LLM-driven **script generation** step *before* audio generation. It
rewrites each blog post's original markdown into a natural, spoken-word script
that a text-to-speech voice reads as if a person were reading a book aloud or
explaining the post to a friend. Audio is then generated from that script
instead of the raw plain text.

## Pipeline Placement

`package.json` build script today:

```
astro build → generate-epub.mjs → uv run python main.py → validate-xml.mjs
```

`astro build` writes `docs/<slug>/index.tts` (plain text) via
`src/pages/[slug]/index.tts.ts`. `main.py` reads those files and generates audio.

New flow inside `main.py` — no change to `package.json` ordering:

```
uv run python main.py
   ├── generate_tts_scripts(slug_filter)          # NEW — runs first
   │     reads src/content/blog/<slug>.md
   │     → Ollama rewrite
   │     → public/audio/<slug>/index.script.txt
   └── text_to_speech_on_plain_text(slug_filter)  # CHANGED audio source
         reads public/audio/<slug>/index.script.txt (fallback: docs/<slug>/index.tts)
         → index.mp3 / index.opus
```

## Components

### 1. `generate_tts_scripts(slug_filter=None)` (new, in `main.py`)

Iterate `docs/` folders exactly like the existing audio loop (skip entries with
a `.` in the name). For each `<slug>` that has a `docs/<slug>/index.tts`, derive
the source path `src/content/blog/<slug>.md`. This keeps slug discovery identical
to the audio step (drafts are already excluded from the build output) and only
processes posts astro actually rendered. If the source `.md` is missing for a
slug, skip the LLM step for it and let audio fall back to `index.tts`.

1. Read the **original markdown source** `src/content/blog/<slug>.md`. This still
   contains `![alt](path)` image references, `[text](url)` links, code fences,
   and frontmatter (`title`, `description`, `image`, `alt`) — all of which the
   plain `.tts` text has already discarded.
2. Compute the SHA-256 of the source `.md` file.
3. **Cache gate:** skip the LLM call if the hash matches `script-hashes.json`
   **and** `public/audio/<slug>/index.script.txt` already exists.
4. Otherwise, send the markdown to a local **Ollama** server and write the
   returned script to `public/audio/<slug>/index.script.txt`, then record the
   hash in `script-hashes.json`.

`public/audio/<slug>/` is committed to git and copied verbatim by `astro build`
into the output, so the script survives builds and can be inspected/proofread in
version control (same rationale as the existing audio files).

### 2. Ollama client + prompt

- Config via `.env` (`python-dotenv` is already a dependency):
  - `OLLAMA_HOST` — default `http://localhost:11434`
  - `OLLAMA_MODEL` — default `llama3.1`
- Call `POST {OLLAMA_HOST}/api/generate` (or `/api/chat`) with `stream=false`
  using the `requests` library (already a dependency).
- **System prompt** enforces these rules:
  1. **Strip/replace URLs** — never read a URL aloud. Replace a link with natural
     phrasing ("you can find the link on the blog post") or drop it entirely.
  2. **Point to the blog** — where the post references links, images, or extra
     material, tell the listener to visit the blog for the full post and details.
  3. **Describe images verbally** — when the post has an image
     (`![alt](path)` or frontmatter `image`/`alt`), narrate what it depicts from
     the alt text and surrounding context, as if describing it to someone who
     cannot see it.
  4. **Conversational narrator tone** — rewrite as if a person is reading a book
     aloud or explaining the post to a friend. No markdown artifacts, no code
     read verbatim.
  5. **Preserve meaning, don't summarize** — keep the same content and structure;
     make it speakable, don't shorten it into a summary.
- The user prompt supplies the raw markdown (title + body).

### 3. `script-hashes.json` sidecar (new)

Mirrors `audio-hashes.json`: `{ slug: sha256_of_source_md }`. Lives at repo root
outside `docs/` so `astro build` cannot delete it. Gates only the LLM step,
independent of audio regeneration so either can be forced without the other.
Reuse the existing `_load_audio_hashes` / `_save_audio_hashes` pattern
(parameterised on path) rather than duplicating logic.

### 4. `text_to_speech_on_plain_text` change

Audio source selection, per slug:

1. Prefer `public/audio/<slug>/index.script.txt` if it exists.
2. Fall back to `docs/<slug>/index.tts` when no script exists.

Everything else (chunking, Kokoro, mp3/opus encoding, `audio-hashes.json`
gating) is unchanged.

## Failure Handling

The script step must never break the build.

- **Ollama unreachable / model missing / request error:**
  - If `public/audio/<slug>/index.script.txt` already exists (cached), use it.
  - Otherwise log a warning and leave no script for that slug; audio generation
    falls back to `docs/<slug>/index.tts` (current behaviour). Build succeeds.
- Connection failures are caught per-post so one bad post doesn't abort the run.

## Data Flow

```
src/content/blog/<slug>.md ──► generate_tts_scripts ──► Ollama ──► index.script.txt
                                      │                               │
                              script-hashes.json               (committed, survives build)
                                                                      │
docs/<slug>/index.tts ─────────────────────────────(fallback)────────┤
                                                                      ▼
                                            text_to_speech_on_plain_text
                                                                      ▼
                                    public/audio/<slug>/index.mp3 + index.opus
```

## Testing

- Unit-test the Ollama client with a mocked `requests.post` (success, connection
  error, non-200) — assert fallback behaviour and that the prompt contains the
  five rule directives.
- Unit-test the hash gate: skips when hash matches and script file exists;
  regenerates otherwise. Reuse the existing sidecar test style.
- Unit-test audio-source selection: prefers `index.script.txt`, falls back to
  `index.tts`.
- Manual: run `uv run python main.py --slug battery-station` with Ollama up and
  confirm `index.script.txt` reads naturally and the audio no longer speaks URLs.

## Out of Scope (YAGNI)

- No cloud LLM providers, no API-key handling.
- No new web endpoint or UI for scripts.
- No changes to RSS/EPUB or the `.tts` endpoint itself (it remains the fallback).
- No streaming Ollama responses.
