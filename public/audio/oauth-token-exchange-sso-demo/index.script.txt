Imagine you're building a service that needs to call another internal API on behalf of a logged-in user. This is a pretty standard issue when services need to talk to each other as the user. I had Authentik running already, so I went looking for support for token exchange with RFC 8693.

Unfortunately, it doesn't support it. Instead, it's using JWT authentication, which is a legitimate machine-to-machine pattern, but it has a trust assumption that's easy to miss until something goes wrong.

I built a demo repo that runs both approaches side by side - Keycloak with native token exchange and Authentik with the JWT federation workaround. You can see exactly where they diverge.

Let me walk you through this. Imagine a user logs into App A, and App A wants to show how many todos the user has in App B. App A calls App B's API, but it can't just forward the user's token because it has an audience of app-a, and App B will reject it.

You need the IdP to issue a new token for App B, carrying the original user's subject. There are two options here: Authentik or Keycloak.

With Authentik, you're using JWT authentication with client assertion. It works, but it shifts a security assumption that most people never notice. The problem is that the issued token's subject is not the user; it's a service account. And there's no mechanism in this grant path to forward the original user's identity into the token.

So, App A passes the user identity separately when calling App B. That trust is implicit, not cryptographic. A bug or compromise in App A could cause App B to return another user's data. The user identity is not bound to the token; it's just a header that anyone with App A's credentials could fabricate.

The concrete risk here is if App A is ever compromised - an attacker can call App B as any user in the system, and the IdP has no knowledge of who the request is "for". With Keycloak, the IdP is the one asserting the user's identity in the token. App A cannot forge that claim even if it's fully compromised.

Now, let's look at Option B - Keycloak with RFC 8693 token exchange. This has been supported by Keycloak since version 12. The grant type is urn:ietf:params:oauth:grant-type:token-exchange.

In this case, App A posts the user's access token as the subject token and asks for a new one scoped to App B. Keycloak validates the subject token, checks the exchange policy, and issues a new token with an audience of app-b and the original user's subject in it.

With this approach, App A can call App B without any extra headers, and App B can validate the token's audience and read the subject directly from it. This is cryptographic trust; there's no room for error.

Here's a summary of the key differences between these two approaches:

- Authentik - client credentials fallback, broken identity chain
- Keycloak - RFC 8693 token exchange, preserved identity chain

The main takeaway here is that Keycloak just works for RFC 8693. With Authentik, you're trusting App A to tell the truth about who the user is. But with Keycloak, the IdP is asserting it and App B can verify that independently.

You can run this demo by cloning the repo and following the instructions in the README file. It's a good exercise to understand how token exchange works and which approach to choose for your own projects.
