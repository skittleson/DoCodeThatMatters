Securing an ASP.NET WebApi for PCI application tests is no trivial matter. It can take weeks of planning and the app to be analyzed. After reviewing OWASP and other related XSS sites, the web.config was the first to be updated to prevent against attacks like Man-In-The-Middle and Cross-Site Scripting.

The featured image illustrates ASP.NET XSS protection.

In the system.webServer element, you add custom headers. First, you remove the Server, X-Powered-By, X-Frame-Options, X-XSS-Protection, X-Content-Type-Options, Cache-Control, Pragma, Expires, and Content-Security-Policy headers. You also clear anything set by IIS. Then, you add them back with secure settings: X-Frame-Options set to DENY, X-XSS-Protection set to one mode block, X-Content-Type-Options set to nosniff, Cache-Control set to no-cache, no-store, must-revalidate, Pragma set to no-cache, and Expires set to a date far in the past, specifically Sunday, January first, nineteen ninety. Finally, you add a Content-Security-Policy that allows resources from self, unsafe-inline, and data, permits fonts from any source, and allows images from https sources only.

The content security policy was the only one considered less secure due to outside JavaScript and CSS resources the app used, such as Google Analytics and Bootstrap. You can find more details on this in the linked blog post.

Moving on to additional protection, there are two main points. First, apply the AntiXss attribute from the ASP.Net Web API AntiXss Attribute project. Second, use a Website Application Firewall for your apps. Some examples of these firewalls include Amazon WAF, ModSecurity, IronBee, NAXSI, WebKnight, and Shadow Daemon.

Next, let's talk about tools. You should look for open source tools that are commonly used for penetration testing. These tools gave some pretty good insight on where the app needed securing. The specific tools mentioned are nikto2, sqlmap, and OWASP Zap.

Finally, here are the takeaways. First, don't rely on just one layer of protection. Second, use established frameworks. Third, use the same tools as the pen testers.

The references for this post include a Stack Overflow search for XSS, the OWASP main page, an MSDN library article, and the Mozilla Developer Network documentation on CSP.
