Imagine sitting down with a friend who's struggling to secure their ASP.NET WebApi. You'd sit them down, make 'em a cup of coffee, and start explaining...

Securing an ASP.NET WebApi for PCI application tests - that's no trivial matter, my friend. It can take weeks of planning and analysis before you're even close to done. After reviewing all the best practices out there, I started with the web.config. That's where you want to update it first to prevent those nasty XSS attacks.

Now, let me show you what you need to add to your system.webServer element in the web.config file. This is where things get interesting. You'll see a bunch of custom headers that we're going to remove and then add new ones. Think of it like a game of Jenga - you take out the old pieces, and then carefully place the new ones.

When I said "custom headers," I mean exactly that. These are special instructions that tell your web server how to behave when requests come in. We're removing all sorts of things here: Server, X-Powered-By, X-Frame-Options... you get the idea. Then we clear out anything else that might be set by IIS.

Now, don't worry if this looks like gibberish - it's not too complicated once you see how it fits together. We're adding new headers here: X-Frame-Options to deny framing (that means no one can embed your site in another page), X-XSS-Protection to prevent cross-site scripting attacks, and a few others. 

Oh, and let me tell you about Content-Security-Policy - that's the one thing I considered less secure because our app uses outside JavaScript resources from Google Analytics and Bootstrap. But don't worry, it's all about balance.

By the way, have you taken a look at your content security policy? It's crucial in preventing cross-site scripting attacks. You can find more information on it by checking out the blog post.

Now, let's talk additional protection. We've got a few tricks up our sleeve here. First, there's an AntiXss attribute from this fantastic ASP.Net Web API AntiXss Attribute project. You should really check that out - it's been a game-changer for us.

And then there are website application firewalls: Amazon WAF, ModSecurity, IronBee... you get the idea. These guys can be lifesavers when it comes to keeping your app secure.

Of course, we're also going to want to use some tools that'll help us identify vulnerabilities. Nikto2, sqlmap, OWASP Zap - these are all great resources for penetration testing.

But here's what I've learned: don't rely on just one layer of protection. Use established frameworks and the same tools as the pen testers. Trust me, it makes a world of difference.

I'll stop there for now, but do check out the references at the end - especially that link to OWASP, which is like the bible for web security.
