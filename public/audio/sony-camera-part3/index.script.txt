You know, I've been working on modernizing my camera setup, and today I want to share with you how I built a laptop-free tethered camera system using a Raspberry Pi. It's a custom Bluetooth binary protocol that enables fast image transfer and zero-config LAN browsing.

Now, let me give you a quick rundown of the TLDR section: if you're interested in trying this out, you can find the link to the code on my blog post. Just drop the files onto your Raspberry Pi Zero via SSH using a micro USB cable, and you're good to go!

As I mentioned earlier, I've been working on modernizing my camera setup, and this project is a continuation of that effort. I recently acquired a Sony a6400, which has some new features I'm excited about. Before we dive in, let me highlight the biggest changes since my last posts: no SSH into the device, no PlayMemories app updates, and no Android subsystem.

Now, what makes this project different is that we're using additional hardware for on-the-go capture and viewing. We have a Raspberry Pi Zero W with GPS, an external battery, and some small cables. This setup gives us real-time shots, online/offline networking, and convenience.

One of the dead ends I encountered was using the Sony app, which was kind of slow and flaky to get going. I tried various methods, but it might be due to my phone or camera itself. As a result, I had to explore alternative solutions.

Let me walk you through some of the issues I faced: Wi-Fi hotspot mode, USB mass storage, Sony Camera SDK full download, FTP from camera, and gphoto2 with a laptop. While these methods worked, they were not very mobile-friendly.

However, there is one cool feature about gphoto2 in PC Mode – it allows us to use camera events. These events can help us gain insights and determine what to do next. Just don't want to carry a laptop around!

Now, let's dive into the architecture of our setup:

Sony a6400 (PC Remote mode)
        │ USB
        ▼
   gphoto2 --capture-tethered
        │
        ▼
   ~/downloads/NNNNN.jpg
        │
        ├──► Bluetooth RFCOMM server
        └──► Copyparty HTTP server

We have three services: tethered capture, Bluetooth RFCOMM, and copyparty HTTP. We can run all three or any subset, and the camera is always connected via USB in PC Remote mode.

The most fragile part of our system is the tethered capture. Most Sony cameras support PC Mode, which allows gphoto2 to stream real-time events and full images over USB. I rebuilt gphoto2 from source, which takes a while on the Pi but provides more stability at version 2.5.32+. Use `setup.sh` to build from source.

One of the engineering pain points was implementing the Lua monitor pattern, not a systemd Restart trap: gphoto2 exits when the USB link blips; systemd restart is fast enough to collide with the camera still holding the USB device – this would result in an error. To solve this issue, I used a Lua script that maintains one persistent gphoto2 session, kills `gvfs-gphoto2-volume-monitor`, waits 5 seconds, and relaunches. This way, systemd sees a healthy Lua process and never triggers its own restart logic.

Another problem we encountered was sequential filenames: gphoto2's `%Y%m%d%H%M%S.jpg` sorts wrong on mobile. To fix this, I used `--filenumber=N --filename %05n.%C`, which results in `00001.jpg`, `00002.jpg`. The Lua script scans the download directory for the highest number before each session to prevent overwrites after a crash or restart.

Finally, we had an issue with the silent failure trap: gphoto2 writes a temp file to CWD before moving to target filename. By default, systemd's CWD is `/`, which is not writable by the service user. This would cause gphoto2 to exit 0 without writing the photo to the download directory.

Now, let's move on to the USB mode-switching bug – this was the most annoying physical bug in our system. I'll mention that it's a physics issue, not software. When the Sony a6400 falls from PC Control mode (`054c:0caa`) to charging mode (`054c:0994`), gphoto2 can't find the camera in charging mode.

To fix this issue, you can use a short quality cable, powered USB hub, or neither – it's a matter of physics. I've documented some troubleshooting steps on my blog post if you want more information.

Next up is our custom Bluetooth binary protocol. The easiest approach was to use RFCOMM, but that's slower due to compression issues. OBEX/OPP is bloated, and HTTP over Bluetooth PAN/DUN is finicky on both sides. So, I created a simple custom protocol with only 3 bytes of overhead per message: `[1 byte opcode] [2 bytes big-endian length] [payload]`.

Here are some commands in this protocol:

* `0x01 LIST`: no payload, returns `0x81` with newline-delimited filenames
* `0x02 GET`: payload is `<quality byte><filename>`, returns `0x82` with JPEG data
* `0xFE`: error responses with ASCII messages (`FILE_NOT_FOUND`, `BAD_QUALITY`, etc.)

We also have quality tiers (server-side compression using Pillow/LANCZOS) to reduce the size of images over Bluetooth:

| Quality | Resize | JPEG quality | Typical size |
|---------|--------|-------------|-------------|
| 0 | None (original) | — | ~7-10 MB |
| 1 | Half-size | 75 | ~1 MB |
| 2 | 1200px wide | 75 | ~300 KB |
| 3 | 1200px wide | 40 | ~100 KB |

This setup allows us to transfer high-quality images over Bluetooth in under 10 seconds.

This setup has some use cases, such as booths, events, or bigger screens without Bluetooth.

In the lessons learned section, I'll highlight some key takeaways from this project:

* Build gphoto2 from source – distro package is too old; plan for build time
* Use a Lua wrapper, not a raw systemd service – gphoto2 exits; the Lua monitor handles the USB race condition that systemd restarts can't fix
* Set `WorkingDirectory` in your systemd unit – the silent failure is the most expensive bug in this project
* Quality cable, powered hub, or neither – USB mode-switching is physics, not software
* Compress server-side for Bluetooth – Pi has the CPU; use it
* Three bytes beats HTTP on RFCOMM – length-prefixed packets are simpler and faster
* Sequential filenames for mobile – timestamps sort wrong on phone galleries

Finally, let's talk about some potential use cases for this setup:

* Product photography – camera on tripod, Pi on shelf, check phone over Bluetooth. No desk, no laptop cable.
* Wildlife – remote trigger, instant photo availability. You're in the field, not at a desk.
* Events – hands-free capture station. Attendees review on LAN browser, collect originals later.
* Booths and displays – self-contained photo station. Pi hides behind the setup. No laptop footprint = looks like magic.
