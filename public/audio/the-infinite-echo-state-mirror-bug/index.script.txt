The Infinite Echo: A State-Mirror Bug That Applies the Effect but Not the Dedup Key.

At 2 a.m., the hallway started clicking. Not once, but continuously. Two wall switches were flipping each other on and off in a tight loop—a sort of soft relay machine gun in the dark. When I finally checked the logs, the shape of the problem was unmistakable. The log entries showed that one device would receive an "off" command, turn itself off, and then immediately broadcast an "off" signal to its neighbor. The neighbor would do the same thing, and the cycle would repeat forever. Each device applied the incoming change and then re-announced it. If you have ever watched a replication storm, a webhook retry loop, or two event handlers feeding each other, you know this shape. It just doesn't usually live in your hallway wall.

The mechanism here is actually quite general: it involves a state-mirroring protocol where the receiver applies the effect of an action but never updates the deduplication key used to suppress its own echo. Because that specific piece isn't updated, every incoming message looks new to the system. Instead of settling, the system amplifies the signal.

I built this into two light switches using Tasmota and MQTT while setting up a new house's smart-home stack. The switches are Martin Jerry units communicating with my own broker. The goal was to create a virtual three-way: flip either switch, and the other would follow. It worked great for months until a power blip hit. Both devices rebooted and began screaming at each other.

This is how the bug works, how I found it live rather than just guessing, and the one-line fix that finally stopped it.

Let's look at the setup. We had two switches mirrored over MQTT. The basic approach was straightforward: when switch A changes, publish its new state to switch B, and vice versa. To prevent an infinite loop, I used a logic check where the system would only broadcast a change if the new value didn't match the last known value stored in a variable called "VAR1." The theory was that since the command coming back from the partner would match the local "VAR1," it would be ignored by the rule and wouldn't trigger another broadcast. One hop each way, then silence.

It looked correct. It had that satisfying simplicity that makes you feel smart. It also worked for months. But it was still wrong; it just took a power outage to prove it.

When the failure occurred after the power blip, my first instinct was a classic boot race: both devices coming online at once with empty variables. I tried to fix this by adding a "boot-settle" guard—disabling the mirror rule for ten seconds after a reboot before re-enabling it. Another power blip happened, and they looped again. The band-aid didn't hold.

That was the moment to stop guessing and actually look. Instead of trying to out-think the problem with more theories, I increased the log levels and watched both devices live while I triggered the loop by hand. 

By examining the logs, I learned two things immediately that were more interesting than my boot-race theory. First, my "physical-press" fix wasn't even active; both switches were still running the original flawed logic. It is always important to verify exactly what code is actually running before you try to debug a problem you think you’ve already fixed.

Second, and this was the real breakthrough: the echo guard never actually broke the loop because of how the data was processed. The logs showed that when the peer's message arrived as a direct write to the "Power" command, it flipped the physical relay, but it never updated the "VAR1" variable on the receiving device. Because the value in "VAR1" remained stale, the system saw every incoming message from the partner as a "new" change and re-published it. 

You can see this clearly in a supporting image of the Tasmota console. It shows both the plain ESP chip and the Tuya device firing out "Publish" commands to each other on every single hop, effectively feeding the loop back into itself.

The peer's mirror command arrived as a direct write. It flipped the relay, but since it didn't update the local variable, the rule triggered every single time. My guard only worked for a single, isolated, local toggle. Under any sustained back-and-forth, it was useless. And even a debounce wouldn't have helped; it would have just slowed down the "machine gun" effect rather than stopping it.

I then tried to pivot my strategy by mirroring only on the physical button press, reasoning that an MQTT-driven relay change shouldn't count as a manual interaction. This should have been structurally loop-proof. However, the logs proved this was impossible for certain hardware. One of the switches is a Tuya device where the main chip communicates with a secondary microcontroller over serial. On these devices, there is no distinction between a button press and a commanded relay change; they both report as the same event. Because the two types of input were indistinguishable on the hardware level, "only reacting to physical presses" wasn't an option I could implement. It was a dead end.

Once I understood the actual mechanism, the fix was almost embarrassingly simple. The problem was that an incoming mirror command updated the relay but ignored the variable_one check. To fix this, I changed the flow: instead of letting the peer write to the "Power" command directly, they now send an event. The receiving device then updates its internal variable first, and only after that is done does it move the relay. 

Because the variation is updated before the hardware moves, when the system checks if it needs to broadcast the change, it sees that the new state matches the updated variable and stays quiet. The echo dies after exactly one hop. No timers, no debounce, and no boot guards were required. It works perfectly even on the Tuya device because everything is handled through standard power and MQTT signals.

There is one thing to keep in mind for deployment: before activating this rule, you must ensure both relays are in the same state and that the local variables are synchronized to match. Otherwise, they will start up with different values and immediately begin fighting to synchronize.

I didn't trust a fix I hadn't tried to break yet. I tested it by performing six rapid toggles on one switch while recording the logs for both. 

A second image shows this successful test. It displays six clear follow actions with no unintended echoes. In each instance, the receiving switch successfully processed the sync event and updated the relay, but because its internal variable was updated first, it never triggered a redundant "Publish" command. The path for the echo was completely closed. When I stopped pressing the button, everything stayed quiet, and both switches remained in the same state.

There are several key takeaways from this experience:

First, always verify what is actually running before debugging what you think is running. I nearly spent hours trying to fix a bug that had already been corrected in the code but not yet deployed.

Second, go watch the thing. Ten minutes of looking at live logs is far more effective than an hour of theorizing about boot races. The logs provided both the root cause and revealed a hardware limitation I hadn't even considered.

Third, remember that debounce isn't always the answer. It felt like the right fix because fast presses triggered the loop, but the issue was a logic error in state tracking, not a timing issue. Slowing it down would have only hidden the problem.

Fourth, the smallest correct fix usually comes after you actually understand the underlying mechanism. Everything I tried before that was just a way to put tape over the cracks.

Finally, this category of bug is common. Any time a node applies an incoming change but fails to update the key used to identify its own echo, a two-party mirror becomes an amplifier. This happens in light switches, in failed CRDT merges, and in webhook retry storms. The fix is always the same: update the deduplication key before you apply the effect.

The hallway is quiet now. For a smart home project, that’s the highest praise there is.
